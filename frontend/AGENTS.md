# Frontend Conventions

## Folder Structure

```
src/
├── app/                  # Application layer
│   ├── components/       # App-shell components (Layout, Sidebar, AuthGuard)
│   ├── routes/           # Page-level route components
│   ├── stores/           # RootStore + StoreContext (application wiring)
│   ├── App.tsx           # Root component (composes Provider + Router)
│   ├── provider.tsx      # Global providers (Store, Theme, Localization, Notifications, ErrorBoundary)
│   ├── router.tsx        # BrowserRouter + route definitions
│   ├── theme.ts          # MUI theme configuration
│   └── index.css         # Global CSS reset
│
├── api/                  # Auto-generated API clients (do not edit manually)
│   ├── apiConfig.ts      # API configuration, auth token handling, client instances
│   ├── http-client.ts    # Base HTTP client (generated)
│   ├── data-contracts.ts # API DTOs (generated)
│   └── *.ts              # Per-resource API clients (generated)
│
├── shared/               # Shared code used across features
│   ├── components/       # Reusable UI components (ErrorBoundary, NotificationProvider, ConfirmDialog)
│   ├── hooks/            # Shared hooks (useRegexTest)
│   ├── stores/           # Shared store utilities (storeUtils)
│   ├── utils/            # Utility functions (regex)
│   ├── constants.ts      # App-wide constants
│   └── index.ts          # Barrel export
│
├── features/             # Feature modules
│   ├── auth/             # Authentication (AuthStore)
│   ├── transactions/     # Transaction management
│   ├── categories/       # Category & rule management
│   └── email/            # Email config & parsing rules
│
└── main.tsx              # Entry point
```

## Dependency Hierarchy

```
app/ → features/, shared/, api/
features/ → shared/, api/, other features/ (via barrel exports only)
shared/ → api/
api/ → (standalone)
```

**Rules:**
- `shared/` must NOT import from `features/` or `app/`.
- `features/` may import from other features only through their barrel export (`index.ts`).
- `app/` can import from everything.

## Feature Structure

Each feature follows this structure (omit folders that aren't needed):

```
src/features/<feature-name>/
├── components/           # Feature-scoped components
│   └── ComponentName/
│       ├── ComponentName.tsx
│       ├── ComponentName.module.less    # (if custom styling needed)
│       └── ComponentNameStore.ts        # (if view-level store needed)
├── hooks/                # Feature-scoped hooks
├── stores/               # Feature state stores (MobX)
├── types/                # Feature-scoped TypeScript types
├── utils/                # Feature-scoped utilities
├── assets/               # Feature-scoped static assets
└── index.ts              # Barrel export (public API)
```

## Component Structure

Each component lives in its own folder matching the component name:

```
src/features/<feature>/components/<ComponentName>/
├── ComponentName.tsx
├── ComponentName.module.less    # Only if custom styling is needed
└── ComponentNameStore.ts        # Only if a view store for the component is needed
```

## Import Conventions

- Use the `@/` path alias for cross-module imports: `import { useStore } from "@/app/stores/StoreContext"`
- Within the same feature, relative imports are acceptable but `@/` is preferred for consistency.
- Cross-feature imports must go through barrel exports: `import { CategorySelect } from "@/features/categories"`

## State Management

- **MobX** for state management with `makeAutoObservable`.
- `RootStore` (in `app/stores/`) aggregates all feature stores.
- `StoreContext` provides the store tree via React context; access with `useStore()`.
- `storeUtils.ts` in `shared/stores/` provides `withLoading` and `extractErrorMessage` helpers.

## API Layer

- API clients in `src/api/` are auto-generated by `swagger-typescript-api` from the backend Swagger spec.
- `apiConfig.ts` is the only manually maintained file in `api/`. It handles token storage, refresh logic, and creates API client instances.
- Regenerate with: `npm run generate-api`

## Styling

- Uses MUI (Material UI) component library with Emotion for CSS-in-JS.
- Theme defined in `app/theme.ts`.
- Prefer MUI's `sx` prop for styling. Use `.module.less` files only when MUI's styling is insufficient.
